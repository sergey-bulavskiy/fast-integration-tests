# Дизайн: демо-тесты «почему не подходит Mock / InMemory / SQLite»

**Дата:** 2026-05-30
**Статус:** дизайн утверждён, готов к плану реализации

## Цель

Для доклада нужен набор тестов, наглядно показывающий, **почему фейковые
подмены реальной БД не годятся**. Каждый тест демонстрирует конкретную проблему,
которая на реальном PostgreSQL не возникает (или ловится), а на фейке —
проскакивает молча, даёт неверный результат или, наоборот, маскирует запрос,
который реальная БД отвергла бы.

Рассматриваемые подмены:

1. **Mock Repository** (Moq) — репозиторий подменяется целиком.
2. **EF Core InMemory** — провайдер без реляционных инвариантов.
3. **EF Core SQLite** — другой диалект и слабая типизация.

## Форма подачи

- **Отдельные классы-демонстрации на каждый фейк** (не матрица «тест × N бэкендов»).
  Каждый тест читается как самостоятельная история.
- **Красный на фейке.** Тест ассертит то, что гарантирует PostgreSQL, и на фейке
  падает — живое красное на экране, максимально убедительно для зала.
- **Docker не нужен.** Фейки работают в памяти; проект запускается мгновенно
  прямо на ноутбуке во время доклада.
- **Единая red-форма даже для «обратных» случаев** (запрос зелёный на фейке, но
  падающий на Postgres): ассертим `Assert.Throws<...>(() => query.ToList())`.
  На Postgres бросает (был бы зелёным) — на фейке не бросает — `Assert.Throws`
  падает — красный. Так весь проект единообразно красный и Docker-free.

## Структура проекта

Один проект `tests/FastIntegrationTests.Tests.Demo.WhyRealDb/` (xUnit, как весь
репозиторий) с тремя папками: `Mock/`, `InMemory/`, `Sqlite/`.

- Ссылки: `FastIntegrationTests.Application`, `FastIntegrationTests.Infrastructure`.
- Пакеты: `Moq`, `Microsoft.EntityFrameworkCore.InMemory`,
  `Microsoft.EntityFrameworkCore.Sqlite`.
- **Без** Testcontainers и Npgsql-провайдера в рантайме (Npgsql приедет
  транзитивно через Infrastructure, но провайдер не используется).

### Сборка схемы у фейков — `EnsureCreated()`, не миграции

Ключевой момент и сам по себе урок: `EnsureCreated()` строит схему из EF-модели
(конфигурации `IEntityTypeConfiguration`), а raw-SQL миграции (триггеры, FTS,
materialized views, check-constraints, заданные сырым SQL) **не применяются**.
Отсюда напрямую растёт SQLite-хит про отсутствующий триггер.

Проверено: конфигурации EF чистые — только `HasColumnType("decimal(18,2)")`,
`HasMaxLength`, `HasIndex().IsUnique()`, ключи и связи. Postgres-специфики
(tsvector, jsonb, computed columns) в EF-модели нет — она вся в raw-SQL
миграциях. Значит `ShopDbContext` + `EnsureCreated()` собирается и под InMemory,
и под SQLite.

### Базовые классы

- `InMemory/InMemoryDemoBase` — `CreateContext()`: `ShopDbContext` на
  `UseInMemoryDatabase(Guid.NewGuid())` + `EnsureCreated()`. Свежий контекст на
  тест, фабрики репозиториев (`new ProductRepository(ctx)` и т.д.).
- `Sqlite/SqliteDemoBase` — держит открытый
  `SqliteConnection("DataSource=:memory:")` весь тест (иначе `:memory:` БД
  исчезает после закрытия соединения), `UseSqlite(conn)` + `EnsureCreated()`,
  `IDisposable` закрывает соединение. Те же фабрики репозиториев + хелпер для
  `FromSqlRaw` (диалектные хиты).
- `Mock/` — без БД. Хелпер `MockedServices` собирает сервис с
  `Mock<IXxxRepository>`; каждый тест настраивает моки «наивно», как сделал бы
  разработчик.

## Курированные хит-листы

### Mock (Moq) — «у мока нет состояния и он не выполняет запросы» (5 хитов)

Подменяем `IProductRepository` / `IOrderRepository` / `ICustomerRepository`,
тестируем сервисный слой. Настраиваем мок наивно, ассертим реальный исход.

1. **Нет read-after-write (нет общего состояния между вызовами).**
   `service.UpdateAsync(id, …)` затем `service.GetByIdAsync(id)` — ассертим, что
   изменение видно. Два вызова мока — независимые стабы, изменение «теряется» →
   красный. Урок: разработчик уверен «я протестировал Update», а мок ничего не
   persist'ил — проверка фиктивна.
2. **Дрейф формы запроса (Include-контракт).** `OrderService` рассчитывает, что
   `GetByIdWithItemsAsync` отдаёт `Items` через `.Include(...)`. Наивный мок
   отдаёт «правильную» форму руками. Если в реальном репозитории `.Include`
   потеряется — прод вернёт пустые `Items`, итоги/DTO поедут, а мок-тест
   останется зелёным навсегда. Red-форма: ассертим контракт формы, который
   должен давать реальный запрос; мок застаблен правдоподобно-неверно. Урок:
   мок = твоё представление о запросе, а не запрос.
3. **Запросы не выполняются.** Сервис отдаёт список из репо; мок возвращает
   заранее заданный список → реальный баг в `WHERE/ORDER BY` репозитория никогда
   не отработает. Assert семантики фильтра → красный.
4. **Нет identity/дефолтов БД.** Мок `AddAsync` возвращает сущность с `Id=0` (БД
   не присвоила ключ). Assert `Id>0` после создания → красный.
5. **Нет транзакций/атомарности.** Многошаговая операция должна откатиться при
   сбое; мок не откатывает. Assert неизменности состояния → красный.

### InMemory (EF Core InMemory) — «провайдер без БД-инвариантов» (6 хитов)

Настоящий `ShopDbContext` + `EnsureCreated()`, провайдер ничего не enforce'ит.

1. **UNIQUE не работает** — два customer с одним email сохраняются.
   Assert throws → красный.
2. **FK не работает** — `OrderItem` с несуществующим `ProductId` сохраняется.
   Assert throws → красный.
3. **Cascade/Restrict не работает** — удаление `Product` с `OrderItem` проходит
   молча. Assert restrict → красный.
4. **Транзакции — no-op** — после исключения внутри транзакции изменения не
   откатываются. Assert отката → красный.
5. **Raw SQL / провайдер-specific** — `ExecuteSqlRaw`/`FromSqlRaw` (ILIKE) не
   поддерживается InMemory. Assert работы запроса → красный (throws).
6. **Не транслируемый предикат (зелёный на фейке → падает на Postgres).**
   Запрос с client-eval-предикатом (`StringComparison.OrdinalIgnoreCase`-перегрузка
   либо локальный метод в `Where`). InMemory тихо считает в памяти (зелёный =
   опасная иллюзия), Postgres бросает `could not be translated`. Red-форма:
   `Assert.Throws<InvalidOperationException>(() => query.ToList())` — на InMemory
   не бросает → красный. Урок: «InMemory молча выполняет запрос, который
   настоящая БД отвергает».

> Прежний хит «MaxLength не enforce'ится» убран — поведение EF тут зыбкое.

### SQLite (EF Core SQLite, `:memory:`) — «другой диалект и семантика» (6 хитов)

Важно: SQLite **enforce'ит** UNIQUE и FK (FK при `PRAGMA foreign_keys=ON`, EF
включает по умолчанию) — поэтому здесь хиты про диалект и расхождение схемы, а не
про целостность.

1. **Регистронезависимый поиск (ILIKE)** — Postgres `ILIKE`/`Contains` vs SQLite
   `LIKE` → другой результат. Assert Postgres-семантики → красный.
2. **Точность decimal** — SQLite хранит как TEXT/REAL → ошибка в
   `SUM`/`ORDER BY price`/денежной арифметике. Assert корректного decimal →
   красный.
3. **Collation/сортировка строк** — `ORDER BY name` даёт другой порядок
   (Unicode/регистр). Assert Postgres-порядка → красный.
4. **TIMESTAMPTZ** — таймзона/`Kind` теряются (SQLite хранит как текст). Assert
   сохранения offset/UTC → красный.
5. **SQL-функции** — `string_agg`/`array_agg`/оконные функции отсутствуют в
   SQLite. Assert работы → красный (throws).
6. **Триггеры/схема из миграций отсутствуют** — `EnsureCreated` не применяет
   raw-SQL миграции (price-history trigger, audit log). Побочный эффект триггера
   не наступает. Assert эффекта → красный. Заодно показывает: «миграции к фейку
   не применяются».

> Дополнительный кандидат (опционально): **GROUP BY по неагрегированной колонке**
> через `FromSqlRaw` — зелёный на SQLite, Postgres бросает
> `column must appear in the GROUP BY clause`. Red-форма через `Assert.Throws`.
> Включить, если позволит время.

## Конвенции

- **Подробный XML-doc на каждый тест-метод и тест-класс**, рассчитанный на junior
  девелопера. Единая 5-блочная структура комментария к тесту:
  1. **Что проверяем** — бизнес-сценарий простыми словами.
  2. **Что гарантирует PostgreSQL** — почему на реальной БД зелёный.
  3. **Что делает фейк не так** — конкретный механизм (нет состояния / нет FK /
     client-eval / диалект).
  4. **Почему тест красный** — какой именно assert падает и почему.
  5. **Зелёный эквивалент** — ссылка на аналог в `Tests.IntegreSQL/...`.
  Это половина ценности проекта для доклада — комментарии читаются как
  мини-методичка.
- `[Trait("Category","Demo")]` на каждом тесте (на случай появления `.sln`/CI —
  отфильтровать `Category!=Demo`).
- Тесты **намеренно красные** на фейке — ожидаемое состояние, фиксируется в README.
- Говорящие имена:
  `InMemory_DoesNotEnforce_ForeignKey_OrderItemWithGhostProduct`,
  `Sqlite_DecimalPrecision_SumDivergesFromPostgres`,
  `Mock_HasNoState_ReadAfterWriteIsLost`.

## Открытые вопросы / риски

- **decimal под SQLite** даёт EF-warning при сравнениях — это подыгрывает хиту о
  точности, но возможны неожиданные исключения при `ORDER BY`/`SUM`. На этапе
  реализации проверить фактическое поведение и оформить ассерт под него.
- **Хиты про raw-SQL функции** (`string_agg`, ILIKE) требуют, чтобы в реальном
  коде/репозиториях был метод, использующий эти конструкции. Сейчас репозитории
  простые (`Contains`, `==`). На этапе плана решить: добавлять ли в демо
  собственные запросы через `FromSqlRaw`/LINQ прямо в тестах, или это нормально —
  тесты сами строят запрос к `ShopDbContext`. Решение: тесты строят запрос сами,
  доменный код не трогаем.
- **Client-eval на InMemory**: убедиться, что выбранный предикат действительно
  считается на клиенте у InMemory и действительно не транслируется реляционным
  провайдером (поведение зависит от версии EF Core — зафиксировать на текущей).
