# Паузы вокруг старта контейнеров в Respawn и IntegreSQL менеджерах

## Симптом

Бенчмарк упал в Сценарии 1, точка `[14/42] Respawn / migrations / m=117 s=12 t=12`,
сразу после `[13/42] IntegreSQL / m=117`. Все 2316 тестов в прогоне свалились
с одинаковой ошибкой:

```
Docker.DotNet.DockerApiException : Docker API responded with status code=InternalServerError,
response={"message":"failed to set up container networking: driver failed programming
external connectivity on endpoint testcontainers-ryuk-cb0f061b-5f3b-466a-ba7d-bf304f453369
(...): failed to bind host port 0.0.0.0:35028/tcp: address already in use"}
```

Стек у всех — `RespawnFixture.InitializeAsync()` → `RespawnContainerManager.StartAsync()` →
`PostgreSqlContainer.StartAsync()` → внутри Testcontainers SDK падает старт Ryuk-контейнера.
Время до падения — 2.5s, типичный признак ошибки на этапе инициализации, а не теста.

Один и тот же session ID Ryuk и один и тот же порт `35028` у всех 2316 ошибок означают,
что внутри SDK многократно дёргается старт одного и того же Ryuk: либо он застрял
в «полу-стартовом» состоянии и SDK ретраит, либо порт реально занят чужим (стейл) сокетом.

## Первопричина

### Что происходит между прогонами

`BenchmarkRunner` запускает `dotnet test` отдельным процессом для каждой точки данных.
Между точками вызывается `WaitForRyukToStop` — опрос `docker ps --filter name=testcontainers-ryuk`
с шагом 500ms, пока контейнер не исчезнет (таймаут 60s).

Проблема: `docker ps` показывает «Ryuk-контейнера нет» сразу после `docker stop`, но
очистка сетевых правил Docker daemon делает **асинхронно после удаления контейнера**:

- iptables NAT-правила для port forwarding
- освобождение IP в подсети bridge
- освобождение зарезервированных хост-портов

Эта зачистка длится ещё несколько секунд после исчезновения контейнера из `docker ps`.
Если новый процесс стартует в это окно — его собственный Ryuk пытается забиндить хост-порт,
который ядро/Docker считают занятым стейл-правилом, и получает `address already in use`.

### Почему стало проявляться именно сейчас

В Сценарии 1 точка `m=117` для каждого подхода — самая длинная, потому что миграций
максимум. Это даёт Docker daemon максимальную нагрузку (создание/удаление БД,
открытие соединений, запись в WAL). На мощной машине с 12 потоками у демона физически
больше работы по очистке состояния, чем он успевает делать за такт `WaitForRyukToStop`.

При меньшем `m` или меньшем `t` нагрузка ниже — окно зачистки меньше — race не воспроизводится.

### Почему Respawn упал, а не IntegreSQL

В этой точке IntegreSQL отработал первым (`[13/42]`), оставил «хвост» зачистки. Сразу
за ним стартанул Respawn (`[14/42]`) и попал в это окно. Если бы порядок в раннере
был другим, упал бы тот, кто оказался следом за самым нагруженным предшественником.

То есть проблема — не в самом Respawn, а в **отсутствии барьера между процессами**.

### Что уже работает и почему этого мало

В `ContainerFixture` (Testcontainers per-class, коммит `d3896e1`) уже стоят паузы
до и после `StartAsync` плюс изолированная сеть. Внутри одного процесса между сменой
фикстур это работает.

Но проблема падения — **между процессами**, и in-fixture паузы есть только в
`ContainerFixture`. В `RespawnContainerManager` и `IntegresSqlContainerManager` никаких
пауз нет — менеджеры стартуют контейнер ровно один раз на процесс через `Lazy<Task<>>`,
и этот единственный старт ловит хвост предыдущего процесса.

## Фикс

Добавить пауза-до и пауза-после `StartAsync` во все три места: `RespawnContainerManager`,
`IntegresSqlContainerManager`, и оставить как есть в `ContainerFixture` (там уже есть).
В `RespawnContainerManager` дополнительно добавить изолированную сеть — для паритета
с двумя другими менеджерами и на случай, если в будущем кто-то запустит больше одного
контейнера в Respawn-процессе.

### Назначение каждой паузы

**Пауза ДО `StartAsync` (10s)** — защита от хвоста предыдущего процесса (или предыдущей
фикстуры в случае ContainerFixture). Гарантирует, что:

- iptables NAT-правила прошлого Ryuk полностью убраны ядром
- хост-порты, освобождённые при `docker stop`, ушли из TIME_WAIT
- bridge-подсети возвращены в пул

Эту работу Docker daemon делает асинхронно, и `docker ps` её не отражает.

**Пауза ПОСЛЕ `StartAsync` (10s)** — защита от того, что собственный контейнер
ещё не полностью «прописан» в сети хоста. `await container.StartAsync()` возвращается,
когда Docker рапортует «процесс в контейнере запущен», но NAT-правила и port forwarding
прописываются на хосте ещё ~сотни мс. Если первый коннект из теста уйдёт в это окно —
получит TCP RST до того, как правило вступило в силу.

В Respawn/IntegreSQL менеджерах это менее критично, чем в `ContainerFixture` (Npgsql
в любом случае ретраит при wait-strategy Testcontainers), но симметрично с
`ContainerFixture` — на случай редких race в первых тестах после старта.

### Влияние на бенчмарк

- IntegreSQL: +20s к каждой точке × 14 точек = +4.7 мин общего времени прогона
- Respawn: +20s к каждой точке × 14 точек = +4.7 мин общего времени прогона
- Testcontainers: без изменений (паузы уже были)

**Цифры в отчёте не меняются.** Паузы стоят до и после `Stopwatch.StartNew/Stop`
вокруг `StartAsync` — это значит, что время «container» в `BenchmarkLogger.Write` остаётся
прежним. Удлиняется только wall-clock, а не выводимая в HTML декомпозиция времени.

### Что не делаем

- Не трогаем `WaitForRyukToStop` в `TestRunner.cs` — он полезен сам по себе (дёшево
  отрабатывает, ловит дешёвую часть окна).
- Не трогаем `ContainerFixture.cs` (паузы уже есть). Опционально расширим там
  комментарии, чтобы они были одинаковыми во всех трёх местах.

## Риски и проверка

- Прогон `dotnet build` на всех трёх тест-проектах — должен пройти без warning.
- Линт логики паузы в `Lazy<Task<>>`: `Task.Delay` внутри single-shot `Lazy` исполняется
  один раз (защищён `LazyThreadSafetyMode.ExecutionAndPublication`), не блокирует
  параллельные фикстуры — они все ждут на одном и том же `_container.Value`.
- Полный прогон бенчмарка не запускаем (как просил пользователь). Достаточно, что
  билд проходит и логика паузы тривиальна.
